# AGENTS.strict.md

## Назначение
Этот файл задаёт **жёсткие правила** для coding agent / Codex при работе над desktop-проектом на **Wails + Vue**.

Агент обязан делать **простые, безопасные, минимальные и кроссплатформенные** изменения.

---

## Базовый режим работы
Всегда работать в режиме:

- **simple first**
- **minimal diffs**
- **desktop-aware**
- **cross-platform safe**
- **backend/frontend separation**
- **no unnecessary dependencies**
- **clear contracts**
- **security first**

---

## Что считать истиной
Приоритет правил:

1. безопасность пользователя и данных;
2. корректность поведения;
3. кроссплатформенность;
4. простота сопровождения;
5. минимальность изменений;
6. скорость реализации;
7. красота кода.

---

## Обязательное поведение агента
Перед любым изменением агент должен:

1. определить слой задачи:
   - `frontend`
   - `backend`
   - `bridge`
   - `storage`
   - `platform`
   - `domain`
2. найти **минимальную точку изменения**;
3. проверить, не ломается ли Windows / macOS / Linux;
4. проверить, не ломается ли контракт между Go и Vue;
5. выбрать самое простое решение;
6. явно обработать ошибки и крайние случаи.

Если задача неоднозначна, агент должен выбрать **самое безопасное разумное предположение** и сделать минимальный корректный шаг.

---

## Жёсткие архитектурные границы

### Frontend (Vue)
Frontend отвечает только за:
- UI;
- состояние экрана;
- пользовательские сценарии;
- отображение ошибок, статусов, прогресса.

Frontend **не должен**:
- содержать системную логику ОС;
- напрямую работать с файлами, процессами, секретами;
- хранить критичную бизнес-логику внутри компонентов;
- раздувать глобальное состояние без необходимости.

### Backend (Go + Wails)
Backend отвечает за:
- бизнес-логику;
- файловую систему;
- системные вызовы;
- процессы;
- безопасность;
- хранение и загрузку локальных данных;
- платформозависимое поведение.

Backend **не должен**:
- содержать UI-логику;
- быть привязанным к структуре конкретного Vue-компонента;
- смешивать binding-слой Wails с domain/service логикой.

### Bridge (Wails <-> Vue)
Bridge должен быть:
- явным;
- маленьким;
- типизированным;
- стабильным.

Через bridge запрещено:
- гонять большие неструктурированные объекты;
- тащить случайные внутренние структуры;
- делать “универсальные” методы без сценарного смысла.

---

## Правила по изменениям
Агент должен:
- менять **как можно меньше файлов**;
- не переписывать код без запроса;
- не делать побочный рефакторинг “заодно”;
- не смешивать новую функциональность и большой рефакторинг;
- сохранять существующий стиль проекта.

Каждое изменение должно быть:
- локальным;
- понятным;
- объяснимым;
- обратимо по смыслу.

---

## Запреты
Агенту запрещено без явной необходимости:

- добавлять новые зависимости;
- вводить DI-контейнеры, сложные паттерны, генераторы;
- переносить Electron-подходы в Wails-проект;
- использовать browser-only решения как основу desktop-функций;
- хардкодить пути, shell-команды и OS-specific поведение;
- писать логику “на будущее”, которая ещё не нужна;
- создавать god objects;
- складировать всё в `utils`;
- оставлять небезопасные TODO вместо завершённого решения.

---

## Кроссплатформенность обязательна
Любое решение должно учитывать **Windows / macOS / Linux**.

Обязательные правила:
- не хардкодить пути;
- использовать переносимые API работы с путями и файлами;
- учитывать права доступа и системные директории;
- платформозависимую логику изолировать;
- не рассчитывать на наличие конкретных shell-утилит;
- не предполагать одинаковое поведение ОС.

Если решение неочевидно кроссплатформенно, агент должен предпочесть более консервативный вариант.

---

## Правила безопасности
Агент обязан:

- не хранить секреты в коде;
- не логировать чувствительные данные;
- не предлагать опасные shell-команды без крайней необходимости;
- проверять ошибки файловых и системных операций;
- аккуратно работать с путями, временными файлами и пользовательскими данными;
- выбирать безопасное поведение по умолчанию.

Особое внимание обязательно для:
- токенов;
- ключей;
- файлов пользователя;
- запуска внешних процессов;
- системных разрешений.

---

## Правила по данным
Доступ к локальным данным должен идти через отдельный слой.

Нельзя:
- разбрасывать чтение/запись конфигов по всему проекту;
- хранить формат данных без учёта миграций;
- читать и писать пользовательские файлы без явной обработки ошибок.

Нужно:
- держать формат хранения простым;
- думать об обратной совместимости;
- контролировать создание и очистку временных файлов.

---

## Правила по Vue
Использовать:
- Vue 3;
- Composition API;
- маленькие composables;
- маленькие компоненты;
- явную обработку `loading / empty / error / success`.

Избегать:
- тяжёлого глобального store без причины;
- лишних watchers;
- смешивания UI и бизнес-правил;
- скрытых побочных эффектов в компонентах.

Предпочитать:
- `computed` вместо лишних `watch`;
- явные async-вызовы;
- переиспользуемые презентационные компоненты;
- простую структуру `pages / features / components / shared`, если она уже есть или реально помогает.

---

## Правила по Go
Использовать:
- простые структуры;
- сервисы с понятной ответственностью;
- явную обработку ошибок;
- `context` там, где есть долгие операции или отмена;
- изоляцию platform/storage/domain/wailsapi логики.

Избегать:
- лишних интерфейсов без пользы;
- ненужного параллелизма;
- утечек goroutines и ресурсов;
- смешивания бизнес-логики с Wails binding-слоем.

---

## Правила по ошибкам
Ошибки нельзя проглатывать.

Нужно:
- возвращать ошибки на подходящем уровне;
- добавлять контекст к техническим ошибкам;
- показывать пользователю понятные сообщения;
- не раскрывать чувствительные детали в UI;
- различать recoverable и fatal сценарии.

---

## Правила по производительности
Нужно:
- не блокировать UI долгими операциями;
- не таскать большие объёмы данных через bridge;
- не создавать лишнюю реактивность;
- не плодить фоновые процессы без контроля;
- инициализировать тяжёлые вещи только когда это реально нужно.

---

## Правила по зависимостям
Новая зависимость допустима только если:
1. без неё решение заметно хуже;
2. стандартных возможностей недостаточно;
3. зависимость зрелая и поддерживаемая;
4. она не создаёт лишний вес и кроссплатформенные риски.

По умолчанию агент должен **не добавлять зависимости**.

---

## Правила по тестам
Тесты приоритетно писать для:
1. backend business logic;
2. сервисов;
3. преобразования данных;
4. критичных пользовательских сценариев;
5. исправлений багов.

Не писать тесты:
- на случайные детали реализации;
- на хрупкую внутреннюю структуру;
- ради количества.

---

## Формат ответа агента
При выполнении задачи агент должен отвечать компактно и структурно:

1. что понял;
2. где будет изменение;
3. какие файлы меняет;
4. что именно сделано;
5. какие есть риски / что проверить.

Без воды. Без длинных теоретических отступлений. Без переписывания лишнего.

---

## Формат предлагаемых изменений
Если агент предлагает реализацию новой фичи, он обязан указать:

- слой логики;
- минимальный набор файлов для изменения;
- контракт frontend/backend;
- кроссплатформенные риски;
- как это проверить.

---

## Что агент должен предпочитать
Предпочитать:
- небольшие сервисы;
- небольшие composables;
- явные DTO;
- простые контракты;
- предсказуемую обработку ошибок;
- код, который легко понять за несколько минут.

---

## Что агент должен избегать
Избегать:
- overly generic abstractions;
- больших универсальных helper-модулей;
- скрытой магии;
- цепочек неявных побочных эффектов;
- сильной связности между frontend и backend;
- случайного усложнения ради “красоты”.

---

## Операционный шаблон
Для любой задачи действовать так:

1. классифицируй слой;
2. найди минимальный diff;
3. проверь кроссплатформенность;
4. проверь контракт Wails <-> Vue;
5. реализуй простейшее рабочее решение;
6. обработай ошибки;
7. кратко перечисли изменённые файлы и результат.

---

## Строгий чеклист перед завершением
Перед завершением задачи агент должен проверить:

- [ ] изменение действительно минимальное;
- [ ] frontend и backend не смешаны;
- [ ] контракт между Go и Vue остаётся понятным;
- [ ] нет хардкода путей и OS-specific предположений;
- [ ] ошибки обработаны явно;
- [ ] новые зависимости не добавлены без причины;
- [ ] код читаемый и поддерживаемый;
- [ ] решение подходит для desktop, а не только для web.

---

## Короткая версия для Codex
- Делай минимальные изменения.
- Не ломай Windows / macOS / Linux.
- Не смешивай Vue UI и Go системную логику.
- Не добавляй зависимости без необходимости.
- Не усложняй архитектуру.
- Делай bridge маленьким и типизированным.
- Ошибки обрабатывай явно.
- Безопасность важнее скорости.
- Понятный код важнее умного кода.
